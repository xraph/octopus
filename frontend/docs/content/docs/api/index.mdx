# API Reference

The Octopus gateway exposes a handful of HTTP APIs alongside the traffic it proxies. They fall
into three groups:

## Admin REST API

The Admin API (served by the `octopus-admin` crate) is the management surface: list and inspect
discovered services, routes, and upstreams; read and reload configuration; and query analytics.
It is documented in full — every endpoint, request, and response — on the Observability page.

It is **not** repeated here. See **[Admin REST API](/docs/observability/admin-api)**.

## FARP Registration API

The FARP (Forge API Gateway Registration Protocol) API is how services tell the gateway they
exist. Services push a manifest to register, send periodic heartbeats, and deregister on
shutdown; the gateway fetches their schemas, generates routes, and serves a federated API surface
(OpenAPI / AsyncAPI / GraphQL) plus Swagger UI and ReDoc.

See the sibling page: **[FARP Registration API](/docs/api/farp)**. For the concepts behind it, see
[Service Discovery & FARP](/docs/farp) and [Schema federation](/docs/farp/schema-federation).

## Health and probe endpoints

The gateway serves Kubernetes-style process probes (`/livez`, `/readyz`, `/startz`, plus the
back-compat aliases `/health` and `/healthz`) on the main listen port. These are for liveness,
readiness, and connection-draining — not for management.

See **[Health probes](/docs/observability/health-probes)** for the full path list, status-code
semantics, and drain behaviour.

## Metrics endpoint

Prometheus metrics are exposed in text format on the listen port at `/metrics` (with the
`/__metrics` alias):

```bash
curl http://localhost:8080/metrics
```

<Callout type="info">
  The metrics path is fixed to `/metrics` (and `/__metrics`). See
  [Metrics](/docs/observability/metrics) for the exact metric names Octopus emits and how to
  scrape them.
</Callout>
