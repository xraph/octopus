# CORS

Octopus ships a complete Cross-Origin Resource Sharing (CORS) middleware
(`octopus_middleware::Cors`) that handles preflight `OPTIONS` requests and adds
`Access-Control-*` headers to responses. The configuration schema for it exists
both globally (top-level `cors`) and per-route (`routes[].cors`).

<Callout type="error" title="CORS is not enforced by the gateway today">
  The `Cors` middleware is **not** part of the runtime's global chain, and the
  runtime never reads the top-level `cors` config. As a result:

  - **No CORS headers** are added to proxied responses.
  - **No preflight** `OPTIONS` response is generated by CORS logic. (The auth
    gateway *skips* `OPTIONS` for auth purposes, but nothing produces CORS
    preflight headers.)
  - The top-level `cors` (`CorsGlobalConfig`) field is parsed but **read
    nowhere**.
  - `routes[].cors` is parsed and copied into request extensions as
    `MatchedRouteCors`, which the `Cors` middleware *would* consume — but since
    that middleware isn't in the chain, it has no effect.

  This page documents the schema and the middleware's behavior for completeness
  and for embedded/library use. To enforce CORS today, embed the middleware via
  the [builder](/docs/middleware#using-middleware-programmatically) or terminate
  CORS at an external proxy.
</Callout>

## What the middleware does (when embedded)

When the `Cors` middleware *is* present in a chain, it:

- Resolves an effective config: a per-route `MatchedRouteCors` override if
  present in request extensions, otherwise its own configured defaults.
- For `OPTIONS` preflight: returns `204 No Content` with
  `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`,
  `Access-Control-Allow-Headers`, `Access-Control-Max-Age`, and (if enabled)
  `Access-Control-Allow-Credentials`.
- For other requests: runs the inner chain, then adds
  `Access-Control-Allow-Origin` (and exposed headers / credentials headers) to
  the response.

### Origin handling

- If `allowed_origins` contains `"*"` and credentials are **not** allowed, the
  response uses `Access-Control-Allow-Origin: *`.
- If `allowed_origins` contains `"*"` and credentials **are** allowed, the
  request's own `Origin` is echoed back (since `*` is invalid with credentials).
- Otherwise the request `Origin` is echoed only when it is in `allowed_origins`.

## Configuration schema

These keys validate and parse today even though they are not enforced.

### Per-route: `routes[].cors`

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `allowed_origins` | list&lt;string&gt; | `[]` | Allowed origins (`"*"` for any) |
| `allowed_methods` | list&lt;string&gt; | `[]` | Allowed HTTP methods |
| `allowed_headers` | list&lt;string&gt; | `[]` | Allowed request headers |
| `allow_credentials` | bool | `false` | Allow cookies / `Authorization` |
| `max_age` | int (seconds) | `86400` | Preflight cache lifetime |

```yaml
routes:
  - path: /api/*
    upstream: backend
    methods: [GET, POST, OPTIONS]
    cors:
      allowed_origins:
        - https://app.example.com
      allowed_methods: [GET, POST]
      allowed_headers: [Content-Type, Authorization]
      allow_credentials: true
      max_age: 3600
```

### Global: top-level `cors`

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `allowed_origins` | list&lt;string&gt; | `[]` | Allowed origins (`"*"` for any) |
| `allowed_methods` | list&lt;string&gt; | `[GET, POST, PUT, DELETE, OPTIONS]` | Allowed methods |
| `allowed_headers` | list&lt;string&gt; | `[]` | Allowed request headers |
| `exposed_headers` | list&lt;string&gt; | `[]` | Headers exposed to the browser |
| `max_age` | int (seconds) | `86400` | Preflight cache lifetime |
| `allow_credentials` | bool | `false` | Allow cookies / `Authorization` |

```yaml
cors:
  allowed_origins:
    - https://app.example.com
  allowed_methods: [GET, POST, PUT, DELETE, OPTIONS]
  allowed_headers: [Content-Type, Authorization]
  exposed_headers: [X-Request-ID]
  max_age: 86400
  allow_credentials: false
```

<Callout type="info">
  Because neither block is enforced, the safest interpretation is to treat the
  gateway as **not** handling CORS. Configure CORS at your upstream service or an
  edge proxy until gateway enforcement is wired up.
</Callout>
