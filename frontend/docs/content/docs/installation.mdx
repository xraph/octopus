# Installation

Octopus ships as a single binary, `octopus`, and as a multi-arch container image. The same binary
serves traffic, validates configuration, generates clients, and — when `kubernetes.enabled` is set —
runs the in-process operator.

## Prerequisites

<Callout title="Requirements">
  - **Docker** (for the container image), or **Rust 1.75+** and a C toolchain (to build from source).
  - **Operating system**: Linux or macOS. The container image targets `linux/amd64` and `linux/arm64`.
  - For Kubernetes, a cluster with `kubectl` configured and Helm 3.8+.
</Callout>

## Installation methods

<Tabs items={['Docker', 'From source', 'Kubernetes']}>

<div title="Docker">

### Docker

The image is published to `ghcr.io/xraph/octopus`, built for `linux/amd64` and `linux/arm64`. Its
entrypoint is the `octopus` binary, and it expects a config file mounted at
`/etc/octopus/config.yaml`.

Pull the image:

```bash
docker pull ghcr.io/xraph/octopus:latest
```

Run it, mounting your config and exposing the listen port (`8080`) and the metrics port (`9090`):

```bash
docker run --rm \
  --name octopus \
  -p 8080:8080 -p 9090:9090 \
  -v "$(pwd)/config.yaml:/etc/octopus/config.yaml" \
  ghcr.io/xraph/octopus:latest
```

<Callout type="info">
  Bind the gateway to `0.0.0.0` inside the container (`gateway.listen: "0.0.0.0:8080"`), not
  `127.0.0.1`, or the port will not be reachable from outside the container.
</Callout>

Check the logs:

```bash
docker logs -f octopus
```

See [Docker deployment](/docs/deployment/docker) for environment-variable substitution, the
container health check, and Compose examples.

</div>

<div title="From source">

### Build from source

<Steps>

<Step>
### Install Rust

Octopus requires Rust 1.75 or later. If you don't have it:

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
```
</Step>

<Step>
### Clone the repository

```bash
git clone https://github.com/xraph/octopus.git
cd octopus
```
</Step>

<Step>
### Build the release binary

```bash
make release
```

The optimized binary is written to `target/release/octopus`.
</Step>

<Step>
### Install it on your PATH (optional)

```bash
sudo install -m 0755 target/release/octopus /usr/local/bin/octopus
```
</Step>

</Steps>

</div>

<div title="Kubernetes">

### Kubernetes

On Kubernetes, install the Helm chart published to the OCI registry
`oci://ghcr.io/xraph/charts` as chart `octopus`:

```bash
helm install octopus oci://ghcr.io/xraph/charts/octopus \
  --namespace octopus --create-namespace
```

This is only an outline — the [Kubernetes install guide](/docs/kubernetes/install) covers the CRDs,
the Gateway API prerequisites, raw manifests, and the full Helm values reference.

## Verify the installation

```bash
# Print version information
octopus version

# View help
octopus --help
```

## Create a configuration file

The only required section is `gateway` with a `listen` address. A minimal static config — one
upstream and one route:

```yaml title="config.yaml"
gateway:
  listen: "0.0.0.0:8080"
  workers: 0                 # 0 = one worker per CPU core

upstreams:
  - name: example-service
    lb_policy: round_robin
    instances:
      - id: example-1
        host: 127.0.0.1
        port: 8081

routes:
  - path: /api
    methods: [GET, POST]
    upstream: example-service
    strip_prefix: /api
```

Validate it without starting the server, then run it:

```bash
octopus validate -c config.yaml
octopus serve -c config.yaml
```

<Callout type="info">
  Configuration may be YAML, TOML, or JSON, and `-c/--config` accepts multiple files or a directory
  (merged in order). See the [Configuration reference](/docs/configuration) for the full schema and
  [CLI](/docs/cli) for every command and flag.
</Callout>

## Health and metrics endpoints

Once running, the gateway serves Kubernetes-style health probes on the **listen port** and
Prometheus metrics on the metrics endpoint:

```bash
curl http://localhost:8080/livez      # liveness — 200 while the process is alive
curl http://localhost:8080/readyz     # readiness — 200 when ready to serve traffic
curl http://localhost:8080/startupz   # startup — 200 once the listener has bound
curl http://localhost:9090/metrics    # Prometheus metrics
```

See [Health probes](/docs/observability/health-probes) and [Metrics](/docs/observability/metrics)
for details.

## Upgrading

### Docker

```bash
docker pull ghcr.io/xraph/octopus:latest
docker restart octopus
```

### From source

```bash
cd octopus
git pull
make release
sudo install -m 0755 target/release/octopus /usr/local/bin/octopus
```

### Helm

```bash
helm upgrade octopus oci://ghcr.io/xraph/charts/octopus -n octopus
```

## Troubleshooting

### Command not found

If `octopus` is not found after installing from source, make sure the install directory is on your
`PATH` (or run the binary by its full path, e.g. `./target/release/octopus`):

```bash
echo "$PATH"
```

### Port already in use

If the gateway fails to bind with "address already in use", find the process holding the port or
change `gateway.listen` in `config.yaml`:

```bash
lsof -i :8080
```

```yaml
gateway:
  listen: "0.0.0.0:8081"   # bind a different port
```

For runtime debugging via logs and metrics, see [Observability](/docs/observability). For
task-focused walkthroughs, see the [Guides](/docs/guides). You can also
[open an issue](https://github.com/xraph/octopus/issues).
