# Tracing

This page is deliberately blunt about the state of distributed tracing in
Octopus, so you do not configure something that has no effect.

<Callout type="warn">
  **Distributed tracing is not wired in the runtime today.** A
  [`observability.tracing`](/docs/configuration/observability) config block
  exists (`enabled`, `jaeger_endpoint`), but the gateway does **not** read it to
  build a tracer, does **not** start an OTLP or Jaeger exporter, and does **not**
  propagate trace context to upstreams. Setting `tracing.enabled: true` or a
  `jaeger_endpoint` currently has no runtime effect.
</Callout>

## What actually runs

The gateway initializes the [`tracing`](https://docs.rs/tracing) crate purely
for **structured logging** — a `tracing_subscriber` formatting layer plus an
`EnvFilter`. That is the diagnostic instrumentation that is live today. See the
[Logging](/docs/observability/logging) page for how that is set up and
controlled.

The internal `tracing::info!` / `debug!` / `warn!` spans and events you see in
the logs are emitted by the gateway code, but they are **not** exported to any
distributed-tracing backend. There is no span exporter configured.

## What is present but inert

| Config field | Type | Default | Runtime effect today |
| --- | --- | --- | --- |
| `observability.tracing.enabled` | boolean | `false` | None — not read by the runtime. |
| `observability.tracing.jaeger_endpoint` | string \| null | none | None — no exporter is built from it. |

The default for `enabled` is `false`, so a default deployment is in the correct
state: tracing off, nothing claiming to export.

<Callout type="info">
  The workspace pulls in OpenTelemetry and OTLP crates as dependencies, but they
  are not yet connected to a tracer in the gateway's startup path. If you need
  end-to-end traces today, terminate tracing at your service mesh or upstream
  services rather than expecting the gateway to emit spans.
</Callout>

## Recommendation

- Leave `observability.tracing.enabled: false` (the default).
- For request-level visibility now, rely on [structured logs](/docs/observability/logging)
  and the per-route data in the [Admin API](/docs/observability/admin-api).
- For Prometheus-based latency and request volume, use
  [Metrics](/docs/observability/metrics).
