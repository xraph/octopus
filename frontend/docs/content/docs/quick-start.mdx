# Quick Start

Get Octopus API Gateway up and running in a few minutes.

## Prerequisites

<Callout title="Before you begin">
  Make sure you have [installed Octopus](/docs/installation) — either the `octopus` binary or the
  `ghcr.io/xraph/octopus` container image.
</Callout>

## Basic setup

<Steps>

<Step>
### Create the configuration

Create a file named `config.yaml`. The only required section is `gateway` with a `listen` address;
this config also declares one upstream and one route:

```yaml title="config.yaml"
gateway:
  listen: "0.0.0.0:8080"
  workers: 0                 # 0 = one worker per CPU core

upstreams:
  - name: example-service
    lb_policy: round_robin
    instances:
      - id: example-1
        host: 127.0.0.1
        port: 8081

routes:
  - path: /api
    methods: [GET, POST]
    upstream: example-service
    strip_prefix: /api
```

This configuration:

- Listens on `0.0.0.0:8080` for HTTP traffic.
- Defines an upstream `example-service` with one instance on `127.0.0.1:8081`.
- Routes `/api/*` to that upstream, stripping the `/api` prefix before proxying.

</Step>

<Step>
### Validate, then start the gateway

Check the config parses and is valid, then run it:

```bash
octopus validate -c config.yaml
octopus serve -c config.yaml
```

`serve` and `validate` take one or more `-c/--config` paths (each a file or directory).
`serve` also accepts `-l/--log-level` (`trace`, `debug`, `info`, `warn`, `error`):

```bash
octopus serve -c config.yaml --log-level debug
```

</Step>

<Step>
### Test the gateway

In another terminal, send a request and check the health probes:

```bash
# Proxied to example-service (path becomes /123 after strip_prefix)
curl http://localhost:8080/api/123

# Liveness / readiness probes (served on the listen port)
curl http://localhost:8080/livez
curl http://localhost:8080/readyz

# Prometheus metrics
curl http://localhost:9090/metrics
```

</Step>

</Steps>

## Configuration explained

### Gateway

```yaml
gateway:
  listen: "0.0.0.0:8080"   # address:port to bind — required
  workers: 0               # worker threads; 0 = one per CPU core
  request_timeout: 30s     # per-request timeout
```

`gateway` is the only required top-level section. See [Gateway](/docs/configuration/gateway) for
timeouts, body-size limits, probes, and compression.

### Upstreams

```yaml
upstreams:
  - name: example-service        # unique name, referenced by routes
    lb_policy: round_robin       # load-balancing policy
    instances:
      - id: example-1            # unique instance id
        host: 127.0.0.1          # host or IP
        port: 8081               # port
```

See [Upstreams](/docs/configuration/upstreams) for instance weights, health checks, and circuit
breakers.

### Routes

```yaml
routes:
  - path: /api                   # path pattern to match (must start with /)
    methods: [GET, POST]         # allowed methods; empty means all
    upstream: example-service    # target upstream (must exist)
    strip_prefix: /api           # prefix removed before proxying
```

See [Routes](/docs/configuration/routes) for priority, prefix rewriting, and per-route auth.

## Common scenarios

<Tabs items={['Multiple services', 'Health checks', 'FARP discovery', 'TLS']}>

<div title="Multiple services">

### Route to multiple services

```yaml
gateway:
  listen: "0.0.0.0:8080"

upstreams:
  - name: user-service
    instances:
      - id: user-1
        host: 127.0.0.1
        port: 8081

  - name: order-service
    instances:
      - id: order-1
        host: 127.0.0.1
        port: 8082

routes:
  - path: /api/users
    upstream: user-service
    strip_prefix: /api

  - path: /api/orders
    upstream: order-service
    strip_prefix: /api
```

</div>

<div title="Health checks">

### Add active health checks

```yaml
upstreams:
  - name: api-service
    lb_policy: round_robin
    instances:
      - id: api-1
        host: 127.0.0.1
        port: 8081
    health_check:
      type: http
      path: /healthz
      interval: 10s
      timeout: 2s
      healthy_threshold: 2
      unhealthy_threshold: 3
```

See [Health checks](/docs/concepts/health-checks) for the runtime model.

</div>

<div title="FARP discovery">

### Enable FARP auto-discovery

Instead of listing instances statically, FARP can discover services and generate routes. The
`farp` section is disabled by default:

```yaml
gateway:
  listen: "0.0.0.0:8080"

farp:
  enabled: true
  watch_interval: 5s
  schema_cache_ttl: 5m
  discovery:
    backends:
      - type: kubernetes
        enabled: true
        config:
          namespace: default
          label_selector: "app.kubernetes.io/part-of=myapp"
          watch_interval: 10s
```

See [FARP & discovery](/docs/configuration/farp) and the [FARP](/docs/farp) section.

</div>

<div title="TLS">

### Terminate TLS

```yaml
gateway:
  listen: "0.0.0.0:8443"
  tls:
    cert_file: /etc/octopus/tls/cert.pem
    key_file: /etc/octopus/tls/key.pem
    min_tls_version: "1.3"
```

See [TLS](/docs/configuration/tls) and the [Security](/docs/security) section for mutual TLS and
certificate hot-reload.

</div>

</Tabs>

## Environment variables

Octopus expands `${VAR}` references in the config file before parsing:

```yaml
gateway:
  listen: "${LISTEN_ADDR:-0.0.0.0:8080}"   # default if LISTEN_ADDR is unset

auth_providers:
  primary:
    type: jwt
    secret: "${JWT_SECRET}"                 # required — startup fails if unset
```

`${VAR}` fails to load when `VAR` is unset; `${VAR:-default}` falls back to the literal default.
Set them before running:

```bash
export JWT_SECRET=your-secret-key
octopus serve -c config.yaml
```

## Run with Docker

<Steps>

<Step>
### Create docker-compose.yml

```yaml title="docker-compose.yml"
services:
  octopus:
    image: ghcr.io/xraph/octopus:latest
    ports:
      - "8080:8080"
      - "9090:9090"
    volumes:
      - ./config.yaml:/etc/octopus/config.yaml
```
</Step>

<Step>
### Start it

```bash
docker compose up -d
docker compose logs -f octopus
```
</Step>

</Steps>

See [Docker deployment](/docs/deployment/docker) for the full image reference.

## Graceful shutdown

On `SIGTERM` (or `Ctrl+C`), Octopus flips readiness to 503, waits `gateway.pre_stop_delay` so
deregistration can propagate, then drains in-flight requests within `gateway.shutdown_timeout`
before exiting. See [Graceful shutdown](/docs/deployment/graceful-shutdown).

## Troubleshooting

<Callout type="error" title="Common issues">

**Port already in use** — change `gateway.listen` to a free port:

```yaml
gateway:
  listen: "0.0.0.0:8081"
```

**Connection refused to upstream** — verify the backend is running and the instance `host`/`port`
are correct:

```bash
curl http://127.0.0.1:8081/healthz
```

**Config rejected on startup** — run `octopus validate -c config.yaml`; it reports the exact field
that failed (for example a route referencing an upstream that doesn't exist).

</Callout>
