# Security

Octopus secures traffic at three layers: **authentication** (who is calling), **authorization**
(what they may do), and **transport security** (TLS/mTLS). Each is configured declaratively and
can be applied globally or per route.

## Authentication

Authentication providers are declared once under `auth_providers` and referenced by name from the
global `auth` block or from an individual route. A request is authenticated by the provider its
matched route selects; routes may also opt out with `skip_auth`. Supported provider types:

- **JWT** — validate bearer tokens against a secret or JWKS, checking issuer, audience, and expiry.
- **OIDC** — discover an OpenID Connect provider and validate its tokens.
- **API key** — match keys presented in a header or query parameter.
- **Forward auth** — delegate the decision to an external HTTP service (à la Traefik/NGINX `auth_request`).
- **mTLS** — authenticate clients by their presented certificate.

## Authorization

Once a caller is authenticated, authorization decides whether the request is allowed: role and
scope checks, expression rules evaluated by the embedded **Rhai** engine, or an external **OPA**
policy. Rules can be global or attached to specific routes.

## Transport security (TLS)

Octopus terminates TLS on its listener, supports **mutual TLS** for client authentication, and
**hot-reloads** certificates without dropping connections. On Kubernetes, certificates can be
sourced from Gateway listener Secrets (for example, issued by cert-manager).
